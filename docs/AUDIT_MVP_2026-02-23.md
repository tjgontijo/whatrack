# Auditoria de Código — WhaTrack MVP

**Data:** 2026-02-23
**Realizada por:** Claude Code (Senior Dev Review)
**Escopo:** Segurança, organização de arquivos, tamanho de componentes, arquitetura

---

## Resumo Executivo

O WhaTrack possui uma base sólida com boa separação de responsabilidades, validação com Zod, criptografia de tokens e sistema de rate limiting em três camadas. Entretanto, há **vulnerabilidades críticas de segurança** que bloqueiam o lançamento do MVP, **problemas arquiteturais moderados** com componentes e rotas excessivamente grandes, e algumas falhas de organização de arquivos.

**Veredicto:** Corrigir os itens marcados como CRÍTICO antes de qualquer deploy em produção.

---

## Achados Positivos

- Verificação de assinatura HMAC-SHA256 nos webhooks (comparação timing-safe)
- Criptografia AES-256-GCM para tokens WhatsApp em repouso
- Rate limiting em três camadas (IP / Org / Burst) com Redis
- Distributed locking via Redis para evitar execução concorrente de jobs
- DLQ pattern — webhook logs + retry mechanism
- Security headers configurados (X-Content-Type-Options, X-Frame-Options, X-XSS-Protection)
- Zod para validação de input em boa parte dos endpoints
- TypeScript em modo strict
- Schema Prisma bem estruturado com indexes e relacionamentos corretos
- Camada de serviços sendo utilizada corretamente na maior parte da aplicação

## CRÍTICO — Bloqueia o Lançamento

### 1. Segredos de Produção no Repositório

**Arquivo:** `.env`

O arquivo `.env` com credenciais reais está commitado no repositório. Isso expõe:

- `DATABASE_URL` — string de conexão completa do PostgreSQL (Neon)
- `META_APP_SECRET`, `META_WEBHOOK_VERIFY_TOKEN`, `META_ACCESS_TOKEN` — credenciais da API Meta/WhatsApp
- `OPENAI_API_KEY` — chave da OpenAI com formato válido
- `GROQ_API_KEY` — chave da Groq
- `GOOGLE_API_KEY` — chave do Google
- `REDIS_URL` — string com senha do Redis
- `TOKEN_ENCRYPTION_KEY` — chave AES-256-GCM (compromete todos os tokens criptografados)
- `BETTER_AUTH_SECRET` — secret de assinatura de sessões

**Impacto:** Qualquer pessoa com acesso ao repositório pode acessar o banco de produção, impersonar a aplicação junto à Meta, chamar as APIs de IA sem limite e descriptografar todos os tokens WhatsApp armazenados.

**Ações imediatas:**

1. Revogar e rotacionar TODOS os segredos listados acima
2. Remover `.env` do histórico git: `git filter-repo --path .env --invert-paths`
3. Verificar se `.env` está no `.gitignore` (está, mas foi commitado antes)
4. Criar `.env.example` com valores placeholder
5. Usar Vercel Environment Variables ou similar para produção

### 2. CRON_SECRET com Fallback Trivial

**Arquivos:**

- `src/app/api/v1/jobs/whatsapp-health-check/route.ts` (linha ~24)
- `src/app/api/v1/jobs/webhook-retry/route.ts` (linha ~23)

```typescript
// PROBLEMA
const CRON_SECRET = process.env.CRON_SECRET || 'development-secret'
```

Se `CRON_SECRET` não estiver definido em produção, qualquer pessoa que conhecer o valor padrão pode invocar os jobs de cron manualmente, causando execuções duplicadas ou abusos.

**Correção:**

```typescript
const CRON_SECRET = process.env.CRON_SECRET
if (!CRON_SECRET) {
  throw new Error('CRON_SECRET environment variable must be set')
}
```

### 3. OAuth State Parameter sem Proteção CSRF

**Arquivo:** `src/app/api/v1/meta-ads/connect/route.ts` (linhas ~34-35)

```typescript
// PROBLEMA — organizationId como state não protege contra CSRF
const state = organizationId
```

Um atacante pode interceptar o fluxo OAuth, substituir o `state` pelo ID de outra organização e associar o token de acesso Meta dessa organização à sua própria aplicação.

**Correção:**

1. Gerar token aleatório como `state`: `const state = crypto.randomUUID()`
2. Armazenar no Redis: `redis.setex(`oauth_state:${state}`, 600, JSON.stringify({ organizationId, userId }))`
3. No callback, validar o `state` e recuperar o `organizationId` do Redis
4. Invalidar o token após uso

### 4. Webhook com Assinatura Inválida Retorna HTTP 200

**Arquivo:** `src/app/api/v1/whatsapp/webhook/route.ts` (linha ~138)

```typescript
// PROBLEMA — retorna 200 mesmo com assinatura inválida
if (!isValidSignature) {
  console.error('[webhook] ❌ SIGNATURE VERIFICATION FAILED')
  // ...atualiza log...
  return NextResponse.json({ received: true }) // ⚠️
}
```

Webhooks com assinatura inválida são aceitos silenciosamente. Retornar `200 OK` sem processar é aceitável para não gerar retries da Meta, mas o código atual pode acabar processando o payload caso o fluxo seja alterado.

**Correção:** Tornar explícito o early return antes de qualquer processamento:

```typescript
if (!isValidSignature) {
  await updateLogAsInvalid(webhookLogId)
  return NextResponse.json({ received: true }) // Não processa
}
// ... todo processamento vem depois daqui
```

## ALTO — Resolver Antes do MVP

### 5. Validação de Acesso à Organização Permite Override via Header

**Arquivo:** `src/server/auth/validate-organization-access.ts` (linhas ~69-72)

```typescript
const organizationId =
  extractOrganizationId(request) ?? session.session?.activeOrganizationId ?? undefined
```

A função `extractOrganizationId` lê o header `x-organization-id`. Se um atacante enviar um header com o ID de outra organização, e a validação de membership não for rigorosa o suficiente, pode haver acesso cruzado entre organizações.

**Correção:** Para endpoints sensíveis, usar sempre `session.session.activeOrganizationId` como fonte primária. Só aceitar header `x-organization-id` se explicitamente necessário e após validar membership.

---

### 6. Rota de Ad Accounts sem Validação de Membership

**Arquivo:** `src/app/api/v1/meta-ads/ad-accounts/route.ts` (linhas ~20-45)

```typescript
const session = await getSessionFromRequest(req)
if (!session || !organizationId) {
  return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
}
// ⚠️ Não valida se o usuário é membro da organização
```

Um usuário autenticado pode passar `organizationId` de outra organização e ver os ad accounts dela.

**Correção:** Substituir por `validateFullAccess()` que verifica membership:

```typescript
const access = await validateFullAccess(req)
if (access instanceof NextResponse) return access
```

---

### 7. Rate Limiting Desativado se Redis Cair

**Arquivo:** `src/lib/rate-limit.ts` (linhas ~85-95)

```typescript
catch (error) {
  // Se Redis falhar, permite a requisição
  console.warn(`[RateLimit] ...`);
  return { allowed: true, ... };
}
```

Se o Redis ficar indisponível, o rate limiting é completamente desativado. Isso abre brecha para ataques de força bruta nos endpoints de autenticação e outros.

**Opções:**

- Fail-secure: retornar 503 se Redis não estiver disponível
- Adicionar circuit breaker e alertas de saúde do Redis

---

### 8. Logs de Debug com Dados Sensíveis em Produção

**Arquivo:** `src/services/whatsapp/meta-cloud.service.ts`

```typescript
console.log('[MetaCloudService] DEBUG - AppID Enviado:', process.env.META_APP_ID)
console.log('[MetaCloudService] DEBUG - Token Usado (10 chars):', token.substring(0, 10) + '...')
```

Logs de debug com partes de tokens e IDs de configuração não devem ir para produção. Remover antes do deploy.

### 9. Ausência de Validação de Variáveis de Ambiente na Inicialização

**Impacto:** Falhas ocorrem em runtime ao invés de na inicialização.

**Recomendação:** Criar `src/lib/config.ts`:

```typescript
function requireEnv(name: string): string {
  const value = process.env[name]
  if (!value) throw new Error(`Missing required env var: ${name}`)
  return value
}

export const config = {
  database: { url: requireEnv('DATABASE_URL') },
  auth: { secret: requireEnv('BETTER_AUTH_SECRET') },
  encryption: { key: requireEnv('TOKEN_ENCRYPTION_KEY') },
  // ...
}
```

### 11. Rotas de API com Lógica de Negócio Inline

**Arquivo:** `src/app/api/v1/tickets/route.ts` (527 linhas)

A rota contém filtragem, agregação, parsing de datas e transformação de resposta diretamente no handler. Isso torna o código difícil de testar e impossível de reutilizar.

**Padrão atual (ruim):**

```typescript
export async function GET(req) {
  // validação
  // 10 queries complexas
  // agregação
  // transformação de resposta
  // formatação de datas
}
```

**Padrão correto:**

```typescript
export async function GET(req) {
  const access = await validateFullAccess(req)
  if (access instanceof NextResponse) return access
  const filters = parseFiltersSchema.parse(searchParams)
  const result = await ticketService.list(access.organizationId, filters)
  return NextResponse.json(result)
}
```

### 12. Meta Ads sem Service Centralizado

As operações de Meta Ads estão fragmentadas em várias rotas:

### 13. Cache In-Memory Não Funciona em Multi-Instância

**Arquivo:** `src/app/api/v1/leads/route.ts` (linhas ~28-30)

```typescript
const cache = new Map<string, { ts: number; data: LeadPayload }>()
const CACHE_TTL_MS = 3000
```

Em deploy serverless (Vercel), cada instância tem seu próprio mapa em memória. O cache não é compartilhado entre instâncias, tornando-o ineficaz ou inconsistente.

**Correção:** Usar Redis para cache distribuído.

---

### 14. Parâmetros de Query sem Validação Zod

Vários endpoints leem query params com parsing manual:

```typescript
// Padrão ruim
const page = Math.max(1, parseInt(searchParams.get('page') || '') || 1)
const pageSize = Math.min(100, parseInt(searchParams.get('pageSize') || '') || 20)
```

Deve-se usar Zod para validar e coercing de forma consistente:

```typescript
const paramsSchema = z.object({
  page: z.coerce.number().int().min(1).default(1),
  pageSize: z.coerce.number().int().min(1).max(100).default(20),
})
```

### 16. Sem Proteção CSRF em Operações de Mutação

Endpoints como `PATCH /api/v1/organizations/[id]` e o fluxo OAuth do Meta Ads não possuem validação de CSRF token.

**Recomendação:** Verificar se Better Auth configura `SameSite=Strict` nos cookies. Se não, adicionar middleware de CSRF para rotas que alteram estado.

### 17. Ausência de Audit Log para Operações Sensíveis

As seguintes operações não possuem registro de auditoria:

- Exclusão de organizações
- Alteração de roles de membros
- Atualização de conexões WhatsApp
- Autenticação Meta Ads (OAuth)

O modelo `WhatsAppAuditLog` existe para WhatsApp, mas não há equivalente para operações organizacionais.

---

### 18. Criptografia sem Suporte a Rotação de Chaves

**Arquivo:** `src/lib/encryption.ts`

Usa uma única `TOKEN_ENCRYPTION_KEY`. Se a chave for comprometida, todos os tokens armazenados ficam expostos.

**Recomendação:** Incluir versão da chave no dado criptografado:

```
v1:{iv}:{authTag}:{ciphertext}
```

Assim é possível migrar para nova chave sem descriptografar tudo de uma vez.

## BAIXO — Melhorias Gerais

### 19. Centrifugo Token Endpoint sem Rate Limit

**Arquivo:** `src/app/api/v1/centrifugo/token/route.ts`

Não há rate limiting na geração de tokens WebSocket. Um usuário poderia gerar tokens em volume excessivo.

---

### 20. Race Condition na Criação de Leads

**Arquivo:** `src/app/api/v1/leads/route.ts` (linhas ~128-137)

Duas requisições simultâneas com o mesmo telefone podem passar pela validação e gerar erro de constraint no banco. Usar `upsert` ou tratar o erro de unique constraint de forma elegante.

---

### 21. Tratamento de Erros Inconsistente

Diferentes endpoints retornam formatos de erro diferentes:

```typescript
// Padrão A
{ error: 'Failed', detail: error.message }

// Padrão B
{ error: 'Failed' }

// Padrão C (problemático)
{ received: true }  // retornado mesmo em caso de falha
```

Criar um utilitário centralizado de resposta de erro e padronizar em todos os endpoints.

## Organização de Arquivos

A estrutura geral está bem organizada. Pontos específicos a verificar:

- `src/server/auth/` — diretório `server` dentro de `src` sem estar em `src/lib/`. Considerar mover para `src/lib/auth/server/` para consistência com `src/lib/auth/`
- `src/services/meta-ads/` existe mas as rotas de Meta Ads em `src/app/api/v1/meta-ads/` contêm muita lógica inline. O service deveria absorver essa lógica
- `src/components/ui/sidebar.tsx` com 701 linhas é um componente de UI mas contém lógica de layout complexa — candidato a extração
